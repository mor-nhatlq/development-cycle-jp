02

セキュリティリスク・外部スキル／エージェント利用時

外部ソース（公開レジストリ、GitHub Gist、サードパーティMCPサーバー等）からスキル／エージェントを取り込む際の主要リスク5項目を以下に整理いたします：

  • プロンプトインジェクション（OWASP LLM01） Webスクレイピング結果、ファイル内容、git diff、MCPレスポンス経由の間接攻撃 — エージェントが「前の指示を無視して機密情報を流出させよ」等の命令を含むデータを読み込んでしまうリスクとなります。
  • ツール／関数呼び出しの悪用 改ざんされたスキルが破壊的ツールを呼び出します：rm -rf /、DROP TABLE、ユーザー制御入力を伴うシェル実行などが該当いたします。
  • データ流出 機密データ（.env、本人確認書類、顧客個人情報）がツール呼び出し経由で漏洩するリスクです — 例：攻撃者サーバーへのWebFetch、プロンプトコンテキストへの混入等。
  • サプライチェーン攻撃 未検証リポジトリからのスキル取得・MCPサーバー侵害・依存関係のタイポスクワッティング・悪意あるコードによるスキルのサイレントアップデートなどが懸念されます。
  • ハルシネーションによる逸脱 エージェントが存在しないAPI／パッケージを生成し、悪意ある類似名パッケージ（例：react-router-doom）をインストールしてしまうケースです。ファントム依存関係はサプライチェーン攻撃のリスクに直結いたします。
03

対策メカニズム・リスク低減策

多層防御（Defense in Depth） — 複数の防御層を組み合わせて運用いたします：

  • 🤖 専用セキュリティ監査エージェント 全カスタムエージェント／スキルを専門にスキャンする独立エージェントを配置いたします。プロンプトインジェクションパターン、ロール上書き試行、危険なシェル実行、OWASP LLM Top 10を検出します。プリコミット + CIで実行し、security-scan-agents.mdとして出力されます。
  • 🔒 権限許可リスト方式 .claude/settings.jsonにて、破壊的ツールはデフォルト拒否設定とします。Bash／WebFetch／MCP呼び出しは個別に明示許可が必要です。rm -rf、DROP、生シェル実行は全てブロックいたします。
  • ⚙ ツール実行前フック ツール実行前に入力を検証するフックを配置：プロンプトのサニタイズ、インジェクションパターン検知、シークレットスキャンを実施します。疑わしいパターンに合致した場合は実行を拒否いたします。
  • 📦 サンドボックスモード 高リスクツール（ファイル書込、Webフェッチ、実行系）は、エグレスホワイトリスト + ファイルシステムchrootによる隔離サンドボックス内で実行いたします。ネットワークアクセスは信頼ドメインのみに制限されます。
  • 📜 ソース信頼性確保・ハッシュ固定 スキルは検証済みGitHub組織（許可リスト）からのみ取得いたします。SHAハッシュを固定し自動アップデートは無効化します。新規スキルの導入はテックリード + セキュリティ部門のレビューを必須とします。
  • 📊 監視・監査 改ざん不可のトランスクリプトログ（S3 Object Lockで7年間保管）。異常なツール利用パターンを自動検知し、日次の自動レビュー、四半期ごとの外部ペネトレーションテストを実施いたします。
重要なポイント：セキュリティ監査エージェントは最初に構築するエージェント（他の全エージェントに先行）であり、後続の全エージェントをスキャンする役割を担います。自己ブートストラップ型セキュリティ：「AIがAIを守る」という設計思想となります。

02

TDD（テスト駆動開発）

  • 🔴 RED — 1つの振る舞いに対する最小限の失敗テストを作成
  • ✅ Verify RED — テストが正しい理由で失敗することを確認（ゲート）
  • 🟢 GREEN — テストを通過する最小限のコードを実装
  • ✅ Verify GREEN — 全テスト合格・回帰なしを確認（ゲート）
  • 🔵 REFACTOR — 合格状態を維持しつつコードを整理
  • ⛔ 失敗テストなしでプロダクションコードを書くことは禁止
